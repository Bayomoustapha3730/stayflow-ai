# Naming Conventions

## Purpose

Consistent naming helps contributors understand the system quickly and reduces ambiguity across product, API, and database layers.

## General Rules

- Use names that describe business meaning.
- Prefer full words over abbreviations.
- Avoid generic names such as `Manager`, `Helper`, or `Data` unless the role is precise.
- Keep DTO, service, repository, and entity names aligned with domain language.

## C# Conventions

- Use `PascalCase` for classes, records, interfaces, methods, and public properties.
- Use `camelCase` for local variables and parameters.
- Prefix interfaces with `I`, such as `ICompanyService`.
- Use `Async` suffix for asynchronous methods when the method returns `Task` or `Task<T>`.

## API and DTO Names

- Request DTOs should end with `Request`.
- Response DTOs may end with `Dto` or a more specific suffix such as `SummaryDto`.
- Query DTOs should end with `QueryParameters` when they represent filters, pagination, or search.

## Database Names

- Table names should be plural and domain-specific.
- Foreign keys should use the referenced entity name plus `Id`, such as `CompanyId`.
- Index names generated by EF Core are acceptable unless a custom name improves clarity.

## Multi-Tenant Language

Use `CompanyId` consistently for company-scoped data until a broader tenant model is introduced.
